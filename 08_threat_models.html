    Threat Models That Survive Certification

    Where architecture, determinism, and operational reality converge

    This is where the previous seven articles converge.

    Runtime determinism (Articles 1, 2, 5). Offline PKI discipline (Article 6). Revocation reality (Article 3). CAN constraints (Article 4). Variance control (Article 7). All of these architectural principles exist to solve one problem: certification programs demand coherence between threat model, architecture, and verification.

    Threat modeling is not a security activity. It is an architectural discipline where design decisions are forced to confront operational reality.

    Core Principle: If your threat model is not traceable to architecture, requirements, and safety hazards, it is not a threat model. It is a brainstorming document. Certification does not reward theoretical completeness. It rewards architectural coherence.

    The DER Rejection Pattern

    DERs reject threat models for one fundamental reason: untraceable claims.

    Your threat model document says: "System shall resist spoofed CAN messages."

    DER asks: "Show me the requirement trace."

    You point to: REQ-SEC-042 "CAN authentication required"

    DER asks: "Show me the architecture."

    You show: HMAC-based challenge-response with 28-bit MAC

    DER asks: "Show me the verification."

    You show: Test case TC-AUTH-015 validates replay protection

    DER asks: "Show me the safety integration."

    This is where most threat models fail.

    The threat was modeled. The requirement exists. The code was written. The test passed. But the threat model was never integrated into the safety analysis. The security threat has no linkage to system hazards. The FMEA doesn't reference the authentication failure mode. ARP4761 hazard analysis doesn't include the spoofing scenario.

    DER conclusion: Threat model is not part of the certified design.

    Generic vs Certifiable Threat Models

    Generic threat models enumerate possibilities. Certifiable threat models trace architectural decisions.

    ❌ Generic Threat Model

    • Lists all possible attacks
    • Theoretical completeness
    • STRIDE enumeration
    • No architecture linkage
    • No safety integration
    • Not traceable to requirements

    DER reaction: "This is a security study, not a design artifact."

    ✓ Certifiable Threat Model

    • Scoped to operational context
    • Traceable to architecture
    • Linked to safety hazards
    • Requirements derived and verified
    • FMEA integration
    • Test cases validate mitigations

    DER reaction: "This is integrated into the design baseline."

    The difference: Generic threat models ask "what could happen?" Certifiable threat models answer "what did we decide, and how do we verify it?"

    The Traceability Matrix

    Certifiable threat models live in traceability, not in narrative documents.

    Example: CAN Message Spoofing Threat

    Threat Requirement Architecture Verification Safety Link
    T-CAN-001:
    Spoofed flight control command    		 REQ-SEC-042:
    CAN auth required    		 ARCH-AUTH-03:
    HMAC challenge-response    		 TC-AUTH-015:
    Replay protection validated    		 FMEA-012:
    Auth failure → safe state
    T-CAN-002:
    Replay attack on authenticated message    		 REQ-SEC-043:
    Replay protection    		 ARCH-AUTH-04:
    Counter-based freshness    		 TC-AUTH-016:

    This is certifiable. Every threat traces to a requirement. Every requirement maps to an architectural decision. Every architectural decision has verification. Every failure mode links to safety analysis.

    What makes it certifiable: It's not the threats themselves—it's the traceability. The DER can follow the chain from threat to hazard to mitigation to verification.

    Safety-Security Integration

    Security threats that don't link to safety hazards are ignored in certification reviews.

    The integration requirement: Every security threat must answer: "If this attack succeeds, what is the safety impact?" If the answer is "no safety impact," the threat doesn't require certification effort. If the answer is "potential hazard," the threat must integrate into ARP4761 hazard analysis and FMEA.

    Example integration:

    Security Threat: T-CAN-001 Spoofed flight control command
    Safety Hazard: H-FCS-042 Uncommanded flight control surface movement
    FMEA Entry: Authentication failure → Flight control command rejected → Safe state (surface holds last valid position)
    Severity: Catastrophic (if unmitigated)
    Mitigation: HMAC authentication with deterministic failure mode
    Residual Risk: Remote (mitigated to acceptable level)

    This is how threat models survive certification. The threat isn't analyzed in isolation—it's integrated into the system hazard analysis that already exists for DO-178C/ARP4754A compliance.

    Attack Surface Scoping

    Certifiable threat models scope attack surface to operational reality, not theoretical possibility.

    Generic approach: List every conceivable interface (USB ports, debug interfaces, wireless, network, physical access, supply chain, side channels, timing attacks, power analysis). Result: 200-page threat catalog, most scenarios not operationally relevant.

    Certifiable approach: Define adversary model from operational context. Example: Aircraft maintenance scenario—adversary is technician with physical access to ground tool, network control between tool and aircraft, no access to aircraft interior during flight. Attack surface: Ground tool authentication interface, CAN bus between tool and aircraft LRU, provisioning interfaces. Excluded: In-flight attacks (no adversary access), supply chain (separate certification concern), side channels (not in adversary capability model).

    DERs prefer narrow, defensible scoping over broad theoretical coverage. The question isn't "did you consider everything?" The question is "can you justify your boundaries?"

    Risk Prioritization for Certification

    Not all threats require equal certification effort.

    Certification-aware prioritization: Impact on safety (does this threat map to a system hazard?), architectural complexity (does mitigation require architecture changes after baseline freeze?), verification burden (can we test this within DO-178C structural coverage requirements?), traceability cost (does this create new requirement traces through multiple documents?).

    Example: Authentication timing side channel. Traditional security severity: HIGH (timing attacks can leak key material). Certification priority: LOW (requires sustained physical access during auth, not in adversary model; mitigation would require constant-time crypto, forcing WCET re-analysis across stack). DER assessment: "Acknowledge as residual risk, document threat model boundary, don't retrofit mitigation."

    Example: Replay attack on authenticated command. Traditional severity: MEDIUM (already authenticated, just replayed). Certification priority: CRITICAL (maps directly to safety hazard H-FCS-042, affects flight control surface, requires deterministic failure mode). DER assessment: "This must be mitigated, verified, and traced to FMEA."

    Certification doesn't prioritize by CVSS score. It prioritizes by safety integration and architectural coherence.

    Document Structure That Survives Review

    Certifiable threat models don't need to be long. They need to be traceable.

    Essential elements: Adversary model (who, capabilities, access), attack surface scope (interfaces in scope, explicit exclusions), threat-to-requirement mapping (every threat → requirement ID), architecture decisions (for each threat, what design choice mitigates it), verification approach (how each mitigation is tested), safety integration (threat → hazard → FMEA linkage).

    What to exclude: STRIDE methodology explanation (DER doesn't care about your process), theoretical attack enumeration (focus on scoped threats), generic security principles (don't explain defense-in-depth; show where you applied it), implementation details (threat model is architecture-level, not code-level).

    A 30-page threat model with complete traceability survives review better than a 200-page comprehensive security analysis with no requirement linkage.

    Conclusion: Architectural Coherence

    The title is "Threat Models That Survive Certification."

    The principle: Threat modeling is where all architectural decisions are forced to confront operational reality.

    Articles 1-7 established architectural constraints:

    • Article 1: PKI creates state explosion → Threat model must justify why runtime PKI is excluded
    • Article 2: TLS doesn't fit message buses → Threat model must scope to CAN-appropriate auth
    • Article 3: Runtime revocation is operationally fragile → Threat model must address revocation through provisioning
    • Article 4: CAN has frame budget constraints → Threat model must work within 28-bit MAC limits
    • Article 5: Deterministic failure is required → Threat model must define explicit failure modes
    • Article 6: PKI belongs in provisioning → Threat model must separate trust establishment from runtime auth
    • Article 7: Security must control variance → Threat model must be embedded in architecture, not injected late

    Article 8 synthesizes: The threat model is where these constraints converge. It's not a separate security deliverable. It's the architectural artifact that proves your design decisions are coherent, traceable, and verifiable.

    Final Truth: Certification does not reward theoretical completeness. It rewards architectural coherence. A threat model that cannot be traced to architecture, tested in verification, and linked to safety hazards will not survive review. Threat modeling is not a security activity. It is an architectural discipline.

    For DERs and certification authorities: The question isn't "did you model all possible threats?" The question is "can you trace every security decision through requirements, architecture, verification, and safety analysis?"

    Threat models either demonstrate coherence—or they demonstrate that security was an afterthought.

    There is no middle ground.
